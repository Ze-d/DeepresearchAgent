# V2.1 多 Agent 并发研究 — 方案选型与权衡分析

> **日期:** 2026-06-15 | **关联设计文档:** [v2.1-multi-agent-design.md](2026-06-15-v2.1-multi-agent-design.md)

## 概述

本文档记录了 V2.1 设计过程中每一轮技术选型的备选方案、利弊分析和最终决策依据。共涉及 5 个核心决策点。

---

## 决策 1：整体推进策略

**问题：** V2.1 的 6 个 roadmap 项目（4 个 Agent、并发研究、结果合并、人工审核）如何安排优先级？

### 备选方案

| 方案 | 描述 | 优点 | 缺点 |
|------|------|------|------|
| **A — 速度优先** | 先做并发执行，缩短单次研究耗时 | 用户体验提升明显；技术挑战清晰 | 速度快但质量无保证；人工审核被延后 |
| **B — 质量优先** | 先做多来源交叉验证，提升报告可信度 | 差异化竞争力（vs 普通搜索）；对抗单一来源偏见 | 并发延迟问题不解决；用户等待时间长 |
| **C — 可控性优先** | 先做 Human-in-the-Loop，面向严肃场景 | 满足高要求用户；审核节点是差异化 feature | 审核体验依赖底层研究质量；并发和质量问题仍在 |
| **D — 分阶段推进** | 并发 → 交叉验证 → 人工审核，三者最终都做 | 风险最低；每阶段独立可交付；技术依赖关系自然（并发是基础） | 总工期最长；需要三个 Phase 的设计文档 |

### 决策

**选择 D — 分阶段推进。**

理由：
1. 技术依赖关系清晰：并发执行（Phase 1）是后续交叉验证（Phase 2）和人工审核（Phase 3）的基础
2. 每阶段独立可交付，降低集成风险
3. Phase 1 可立即带来用户体验提升（更短的等待时间）
4. 与 v0-v2 的增量构建风格一致

## 决策 2：多 Agent 并发架构

**问题：** 4 个 Agent 如何在现有 LangGraph 架构中并行执行？

### 备选方案

#### 方案 A：LangGraph Send API 并行扇出（纯 LangGraph）

**做法：** 使用 LangGraph 的 `Send` API 将每个 sub_question 的 source_type 发送到对应的 agent 节点实例，由 LangGraph 运行时自动并行调度。

```
graph.add_node("research_agent", research_agent_node)
graph.add_conditional_edges("plan", fanout_to_agents, path_map=[...])
graph.add_edge("research_agent", "merge")
```

| 优点 | 缺点 |
|------|------|
| 纯 LangGraph，不引入额外框架 | `Send` API 有一定学习曲线 |
| Checkpoint 天然支持（SqliteSaver 直接可用） | 4 个 agent 共用同一个 LLM 配置 |
| LangGraph Studio 可视化调试 | 扇出数固定（4 个类型）不够动态 |
| SSE Streaming 原生支持每个 Agent 的进度事件 | 需要深入理解 LangGraph 内部机制 |
| 与现有 `graph.py` 结构一致，改动最小 | |

#### 方案 B：Python 并发 + LangGraph 节点（混合式）

**做法：** Research 节点内部使用 `concurrent.futures.ThreadPoolExecutor` 或 `asyncio.gather` 并行调用 4 个 Agent 函数，合并结果后作为单次 state update 返回。

```python
with ThreadPoolExecutor(max_workers=4) as pool:
    futures = {pool.submit(paper_agent, sq): "paper", ...}
    results = [f.result() for f in as_completed(futures)]
```

| 优点 | 缺点 |
|------|------|
| 实现简单直接，开发者熟悉 | LangGraph 感知不到并发 — checkpoint 粒度粗（只能存整体完成状态） |
| Agent 数量灵活，可按 source_type 动态创建 | Streaming 困难 — 只能报告"Research 整体完成"，无法逐个 Agent 推送 |
| Graph 结构变化最小（仅一个节点内部改动） | 错误隔离弱 — 一个 Agent 崩溃影响整个 ThreadPool |
| 不需要学习 Send API | 无法中途 resume 单个 Agent（checkpoint 不支持） |

#### 方案 C：工具化 Agent（扩展 tools.py）

**做法：** 将 4 个 Agent 实现为 LangChain Tool，由现有 Research 节点通过 LLM Tool-calling 自行选择调用哪些。

```python
tools = [arxiv_search_tool, github_search_tool, blog_search_tool, docs_search_tool]
llm_with_tools = llm.bind_tools(tools)
```

| 优点 | 缺点 |
|------|------|
| 对现有架构几乎零改动 | LLM 决策不稳定 — 可能漏掉重要来源 |
| LLM 自动决策用哪个工具，"智能"选择 | Tool calling 是顺序的 — 不是真正的并发 |
| 扩展新 Agent = 加一个 Tool，心智负担低 | Agent 逻辑受限于 Tool 接口（只能 search → return） |
| 实现速度最快 | 无法做复杂 Agent 逻辑（多步推理、中间状态） |

### 决策

**选择 A — LangGraph Send API。**

理由：
1. 与现有 StateGraph/Checkpoint/Streaming 体系完全一致，是 LangGraph 生态的"正道"
2. Checkpoint 粒度细 — 每个 Agent 可独立 checkpoint 和 resume
3. SSE Streaming 天然支持 — 每个 Agent 完成时推送 `agent_done` 事件，前端可逐 Agent 更新
4. Send API 学习成本是一次性的，后续 Phase 2/3 都在此架构上自然扩展
5. 方案 B 的 streaming/checkpoint 缺陷在 V2.1 的"可控性"目标下不可接受；方案 C 不是真正的并发

## 决策 3：Agent 内部实现

**问题：** 给定 Send API 扇出架构，4 个 Agent 的内部实现应该是统一代码还是各自独立？

### 备选方案

#### 方案 A：统一节点 + 策略注入

**做法：** 一个 `make_research_agent(llm)` 工厂函数。每个 Agent 实例的差异仅在于 `AgentProfile` 配置（搜索修饰符、system prompt、证据提取指令）。

```
graph.add_node("research_agent", make_research_agent(llm))  # 1个节点，4个实例

AGENT_PROFILES = {
    "paper":  AgentProfile(search_modifiers=["site:arxiv.org", ...], prompt=...),
    "github": AgentProfile(search_modifiers=["site:github.com", ...], prompt=...),
    ...
}
```

| 优点 | 缺点 |
|------|------|
| 代码复用最大化 — 一个函数服务 4 个 Agent | 如需特殊逻辑（如 GitHub API 认证），需 if 分支 |
| 新增 Agent 只需加 Profile 配置 | Profile dataclass 可能随需求膨胀 |
| 测试简单 — 测 1 个节点函数覆盖所有 Agent | 所有 Agent 共享同一 LLM model 和 temperature |
| 公共逻辑（去重、排序）自然集中 | |

#### 方案 B：4 个独立节点 + 共享基类

**做法：** 为每个 Agent 写独立节点函数，继承自 `BaseResearchAgent`，各自定制搜索逻辑。

```
nodes/research/
├── base.py           # BaseResearchAgent
├── paper_agent.py    # PaperResearchAgent — arXiv API
├── github_agent.py   # GitHubAgent — GitHub API
├── blog_agent.py     # BlogAgent — RSS/爬虫
└── docs_agent.py     # DocsAgent — 官方文档搜索
```

| 优点 | 缺点 |
|------|------|
| 每个 Agent 可深度定制（GitHub API、arXiv OAI-PMH） | 代码量约 4x |
| 职责清晰，易于独立测试和调试 | 公共逻辑需在基类中维护（易退化） |
| 扩展不影响其他 Agent | 4 个 make 函数 + 4 个测试文件 |

#### 方案 C：4 个独立 SubGraph

**做法：** 每个 Agent 是完整的 LangGraph StateGraph，有自己的子节点（search → fetch → extract → dedup），在父图中作为 subgraph 节点。

```
paper_graph = StateGraph(AgentSubState)
paper_graph.add_node("search", search_arxiv)
paper_graph.add_node("fetch", fetch_paper)
paper_graph.add_node("extract", extract_evidence)
parent_graph.add_node("paper_agent", paper_graph.compile())
```

| 优点 | 缺点 |
|------|------|
| 最细粒度的 checkpoint 和 streaming | 架构最重 — 5 个 StateGraph |
| 每个 Agent 有独立的状态空间 | SubGraph 间状态传递复杂（需定义子状态映射） |
| 最适合未来扩展为独立微服务 | 过度设计风险高 — Agent 内部未必需要图 |
| | 调试复杂度显著增加 |

### 决策

**选择 A — 统一节点 + 策略注入。**

理由：
1. V2.1 的 4 个 Agent 行为模式高度相似（搜索 → 抓取 → 提取），差异仅在于搜索源和 prompt 偏好
2. 统一节点避免过度抽象 — 5 个 StateGraph（方案 C）对当前需求过度设计
3. 如果未来某个 Agent 需要特殊能力（如 GitHub API 需要 OAuth），可单独升级为方案 B 的独立节点
4. AgentProfile 是清晰且可测试的边界：`(profile, sub_question) → (sources, evidences)`

## 决策 4：Sub-Question 路由分配

**问题：** Plan 节点产生 sub_questions，如何决定每个问题由哪些 Agent 执行？

### 备选方案

#### 方案 A：Plan 节点 LLM 直接分类

**做法：** Plan 节点生成 sub_question 时，LLM 同时输出 `source_types: ["paper", "blog"]`。路由函数按此字段扇出。

```json
{
  "id": "q1",
  "question": "Transformer 架构最新进展",
  "source_types": ["paper", "blog"],
  "search_queries": [...]
}
```

| 优点 | 缺点 |
|------|------|
| LLM 理解问题语义，分类准确 | 依赖 LLM 输出格式稳定（需在 prompt 中约束） |
| 一个 sub_question 可同时发多个 Agent（天然交叉覆盖） | Plan prompt 需要修改和测试 |
| 路由逻辑简单 — 纯字段读取 | 新 source_type 枚举值需要 prompt 同步更新 |
| 几乎零额外 token 成本（在已有 Plan LLM 调用中完成） | |

#### 方案 B：全部广播

**做法：** 不做分类，每个 sub_question 发给全部 4 个 Agent，让每个 Agent 自己判断相关性。

```python
# 每个 Agent 自己判断 sub_question 是否相关
if not is_relevant(sq, profile):  # LLM 快速判断
    return {"evidences": [], "skipped": True}
```

| 优点 | 缺点 |
|------|------|
| 不漏 — 全量覆盖所有来源 | 4x LLM 调用（每个 Agent 都要判断一次） |
| Plan 节点无需改动 | 大量"跳过"浪费 token 和延迟 |
| Agent 自治性强，架构灵活 | 并行度虚高 — 实际有效的 Agent 少 |
| | 用户体验差 — 看到 4 个 Agent 运行但只有 1 个产出 |

#### 方案 C：关键词规则路由

**做法：** 基于 search_queries 中的关键词（arxiv、github.com、paper、docs 等）做规则匹配。

```python
ROUTING_RULES = {
    "paper": ["arxiv", "paper", "论文", "doi", "conference"],
    "github": ["github", "repo", "源码", "implementation"],
    ...
}
```

| 优点 | 缺点 |
|------|------|
| 零 token 开销 — 纯字符串匹配 | 关键词覆盖不全 — 容易漏分和误分 |
| 可预测、可调试 | 维护成本随问题多样性增长 |
| Plan 节点无改动 | 无法处理模糊问题（如"LLM Agent 最佳实践"不包含明确关键词） |
| | 中文/英文关键词需双份维护 |

### 决策

**选择 A — Plan 节点 LLM 直接分类。**

理由：
1. Plan 节点已在调用 LLM，让 LLM 同时输出 `source_type` 几乎零额外成本
2. LLM 对语义的理解远超关键词规则（如"大模型部署方案"会被正确识别为 blog + docs 而非遗漏）
3. 一个 sub_question 可路由到多个 Agent — 这是 Phase 2 交叉验证的前提（同一问题被 paper 和 blog 同时覆盖）
4. 方案 B 的 token 浪费不可接受（4x LLM 判断 × 每个 sub_question）；方案 C 的准确性不足

## 决策 5：Human-in-the-Loop 审核节点位置

**问题：** 人工审核节点应该插入工作流的哪个环节？

### 备选方案

#### 方案 A：Plan 之后（事前把关）

```
Plan → [🛑 HITL] → Research × 4 → Merge → Critique → Final
```

| 优点 | 缺点 |
|------|------|
| 审核成本低 — 评审计划比评审全文快得多 | 人工需要理解子问题才能有效审核 |
| 预防浪费 — 避免 Research 做错方向 | 审核时看不到实际结果，判断依据不足 |
| 适合多轮迭代场景 | 无法发现 Research 阶段产生的质量问题 |

#### 方案 B：Merge 之后（事后把关）✅ 最终选择

```
Plan → Research × 4 → Merge → [🛑 HITL] → Critique → Final
```

| 优点 | 缺点 |
|------|------|
| 信息最充分 — 看到全部 sources/evidences/conflicts | 审核材料多（需通读 sources + evidences） |
| 可标记不足、追加搜索（amend） | 如果审核决定 redo，已完成的 Research 被浪费 |
| 可拒绝低质量 source，直接提升报告质量 | 审核等待时间 = 全部 Research 完成时间 |
| Conflicts 列表提供审核重点 | |

#### 方案 C：Critique 之后（最终把关）

```
Plan → Research × 4 → Merge → Critique → [🛑 HITL] → Final
```

| 优点 | 缺点 |
|------|------|
| 自动化最大化 — 只在最后验证 | Critique 出问题则所有前置工作白费 |
| Critique 评分可作为审核参考 | 审核效率低 — 需要通读完整报告 |
| 适合对自动质量有信心的场景 | 无法在 Research 阶段纠偏 |

### 决策

**选择 B — Merge 之后。**

理由：
1. **信息完整性：** Merge 之后人工能看到所有 Agent 的产出、交叉验证结果和 conflicts，信息最充分
2. **干预时机恰当：** 在 Critique（AI 自动评审）之前介入，人工判断可以补充 AI 评审的盲区
3. **amend 路径自然：** 如果发现 coverage gap，可直接补充搜索查询回到 Research，无需重跑整个流程
4. 方案 A 的"信息不足"和方案 C 的"已太晚"都不如 B 的时机恰当
5. 与 Phase 2 的 MergeSummary（交叉验证结果）天然配合 — MergeSummary 本身就是审核界面

## 决策汇总

| # | 决策点 | A | B | C | D | 选择 | 核心理由 |
|---|--------|---|---|---|---|------|---------|
| 1 | 推进策略 | 速度 | 质量 | 可控性 | 分阶段 | **D** | 技术依赖自然，增量交付 |
| 2 | 并发架构 | Send API | ThreadPool | Tool 化 | — | **A** | Checkpoint/Streaming 原生支持 |
| 3 | Agent 粒度 | 统一+Profile | 4 独立节点 | 4 SubGraph | — | **A** | 行为相似，避免过度抽象 |
| 4 | 问题路由 | LLM 分类 | 全部广播 | 关键词规则 | — | **A** | 零成本准确分类，支持交叉覆盖 |
| 5 | HITL 位置 | Plan 后 | Merge 后 | Critique 后 | — | **B** | 信息最充分，干预时机最佳 |

**核心设计原则贯穿始终：**
- 复用现有基础设施（Checkpoint、Streaming、dedup、ranking）
- 每个 Phase 独立可交付
- 向后兼容 v2 API 和 CLI
- 不过度设计 — 为未来扩展留空间但不提前实现
