4 个 Agent 的内部实现方式？

决定了 Send API 扇出后，每个 Agent 的粒度如何设计？

A

方案A：统一节点 + 策略注入（推荐）

一个 research_agent_node，通过 agent_profile 配置不同搜索策略。4 个实例行为差异仅在于搜索源和 prompt。

# graph.py 只定义1个节点函数
graph.add_node("research_agent", make_research_agent(llm))

# 扇出时注入 agent_profile
def fanout(state):
    profiles = {
        "paper":  {"search_sources": ["arxiv"], "prompt": PAPER_PROMPT},
        "github": {"search_sources": ["github"], "prompt": GITHUB_PROMPT},
        "blog":   {"search_sources": ["tech_blogs"], "prompt": BLOG_PROMPT},
        "docs":   {"search_sources": ["official_docs"], "prompt": DOCS_PROMPT},
    }
    return [Send("research_agent", {"profile": p, "sub_questions": ...})
            for p in profiles.values()]

# 单个节点函数
def make_research_agent(llm):
    def research_agent(state):
        profile = state["agent_profile"]
        # 使用 profile["search_sources"] 搜索
        # 使用 profile["prompt"] 提取证据
    return research_agent

优点

  • 代码复用最大化 — 一个函数服务4个Agent
  • 新增Agent只需加profile配置
  • 测试简单 — 测1个节点覆盖所有Agent

缺点

  • 如需特殊逻辑（如GitHub API），需if分支
  • Profile 容易膨胀
B

方案B：4 个独立节点 + 共享基类

为每个Agent写独立节点函数，继承自 BaseResearchAgent 基类，各自定制搜索逻辑。

# nodes/research/
#   ├── base.py           # BaseResearchAgent 基类
#   ├── paper_agent.py    # PaperResearchAgent(Base) — arXiv API
#   ├── github_agent.py   # GitHubAgent(Base) — GitHub API
#   ├── blog_agent.py     # BlogAgent(Base) — RSS/爬虫
#   └── docs_agent.py     # DocsAgent(Base) — 官方文档搜索

graph.add_node("paper_agent", make_paper_agent(llm))
graph.add_node("github_agent", make_github_agent(llm))
graph.add_node("blog_agent", make_blog_agent(llm))
graph.add_node("docs_agent", make_docs_agent(llm))

优点

  • 每个Agent可深度定制（如GitHub用API、arXiv用OAI-PMH）
  • 职责清晰，易于独立测试
  • 扩展不影响其他Agent

缺点

  • 代码量4x
  • 公共逻辑需在基类维护
  • 4个make函数4个测试文件
C

方案C：4 个独立 SubGraph

每个Agent是完整的 LangGraph StateGraph，有自己的子节点（search → fetch → extract → dedup），然后在父图中作为 subgraph 节点使用。

# 每个 Agent 是一个 mini StateGraph
paper_graph = StateGraph(AgentSubState)
paper_graph.add_node("search", search_arxiv)
paper_graph.add_node("fetch", fetch_paper)
paper_graph.add_node("extract", extract_evidence)
paper_graph.add_conditional_edges(...)

# 父图
parent_graph.add_node("paper_agent", paper_graph.compile())

优点

  • 最细粒度的checkpoint和streaming
  • 每个Agent有独立的状态空间
  • 最适合未来扩展为独立服务

缺点

  • 架构最重 — 5个StateGraph
  • SubGraph 间状态传递复杂
  • 过度设计风险 — Agent内部未必需要图

推荐

方案 A — 统一节点 + 策略注入。
理由：V2.1 的 4 个 Agent 行为模式高度相似（搜索→抓取→提取），差异仅在于搜索源和 prompt 偏好。统一节点避免过度抽象，Phase 2 和 3 也不需要每个 Agent 有独立子图。如果未来某个 Agent 需要特殊能力（如 GitHub API），再单独升级为方案B。
