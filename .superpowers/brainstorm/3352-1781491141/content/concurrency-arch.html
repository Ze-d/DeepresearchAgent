多 Agent 并发架构：哪种实现方式？

核心问题：4 个 Agent 如何在 LangGraph 中并行执行

A

方案A：LangGraph Send API 并行扇出（纯 LangGraph）

使用 LangGraph 的 Send API 将每个 source_type 的子问题发送到对应的 agent 节点，自动并行执行。完全在图内调度。

graph.add_node("plan", plan_node)
graph.add_node("researcher", researcher_node)  ← 单节点，4实例并行
graph.add_conditional_edges("plan", fanout_to_researchers,
    path_map={"paper","github","blog","docs"})
graph.add_node("merge", merge_node)
graph.add_edge("researcher", "merge")

优点

  • 纯 LangGraph，不引入额外框架
  • Checkpoint 天然支持 — SqliteSaver 直接可用
  • LangGraph Studio 可视化调试
  • 与现有 graph.py 结构一致

缺点

  • Send API 有一定学习曲线
  • 4个 agent 共用同一个 LLM 配置
  • 扇出数固定（4个类型）不够灵活
B

方案B：Python 并发 + LangGraph 节点（混合式）

Research 节点内部使用 concurrent.futures.ThreadPoolExecutor 或 asyncio.gather 并行调用 4 个 Agent 函数，合并结果后返回给图。

# research_node 内部：
with ThreadPoolExecutor(max_workers=4) as pool:
    futures = {
        pool.submit(paper_agent, sq): "paper",
        pool.submit(github_agent, sq): "github",
        ...
    }
    results = [f.result() for f in as_completed(futures)]
# 合并 -> return state update

优点

  • 实现简单直接
  • Agent 数量灵活，可按 source_type 动态创建
  • Graph 结构变化最小

缺点

  • LangGraph 感知不到并发 — checkpoint 粒度粗
  • Streaming 困难 — 只能报告整体完成
  • 错误隔离弱 — 一个 agent 崩溃影响全部
C

方案C：工具化 Agent（扩展 tools.py）

不改变图结构，把 4 个 Agent 实现为 LangChain Tool，由现有的 Research 节点通过 Tool-calling 选择调用哪个/哪些。

tools = [
    arxiv_search_tool,    # Paper Agent
    github_search_tool,   # GitHub Agent
    blog_search_tool,     # Blog Agent
    docs_search_tool,     # Official Docs Agent
]
llm_with_tools = llm.bind_tools(tools)
# LLM 自行决定调用哪些工具

优点

  • 对现有架构几乎零改动
  • LLM 自动决策用哪个工具
  • 扩展新 Agent = 加一个 Tool

缺点

  • LLM 决策不稳定 — 可能漏掉重要来源
  • Tool calling 是顺序的 — 不是真正的并发
  • Agent 逻辑受限于 Tool 接口

推荐

方案 A — LangGraph Send API 是 v2.1 正确方向。
理由：与现有 StateGraph/Checkpoint/Streaming 体系一致，LangGraph 原生支持并行执行和细粒度状态追踪。后续 Phase 2（交叉验证）和 Phase 3（人工审核）也能在此架构上自然扩展。

回顾：已确认的决策

优先级：三者兼顾分阶段推进
  1. Phase 1 — 并发研究（速度）：4 Agent 并行扇出
  2. Phase 2 — 交叉验证（质量）：结果合并 + 冲突裁决
  3. Phase 3 — 人工审核（可控性）：Human-in-the-Loop 节点
