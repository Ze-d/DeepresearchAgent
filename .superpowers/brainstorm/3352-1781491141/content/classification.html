Sub-Question 如何分配给 4 个 Agent？

Plan 节点产生 sub_questions，需要一个路由逻辑决定每个问题由哪个 Agent 执行

A

方案A：Plan 节点直接分类（LLM 分类，推荐）

Plan 节点生成 sub_question 时，LLM 同时输出 source_types: ["paper", "blog"]。路由函数按此字段扇出。

# Plan 输出增强
{
  "id": "q1",
  "question": "Transformer架构最新进展",
  "source_types": ["paper", "blog"],  ← 新增
  "search_queries": ["Transformer architecture 2025", ...]
}

# 路由函数
def fanout_to_agents(state):
    sends = []
    for sq in state["research_plan"]["sub_questions"]:
        for st in sq["source_types"]:
            sends.append(Send("research_agent", {
                "agent_profile": AGENT_PROFILES[st],
                "sub_question": sq
            }))
    return sends

优点

  • LLM 理解问题语义，分类准确
  • 一个 sub_question 可同时发多个 Agent（交叉覆盖）
  • 路由逻辑简单

缺点

  • 依赖 LLM 输出格式稳定
  • Plan prompt 需要修改
B

方案B：全部扇出（广播模式）

不做分类，每个 sub_question 发给全部 4 个 Agent，让每个 Agent 自己判断相关性并决定是否产出。

# 每个Agent自己判断 sub_question 是否相关
def research_agent(state):
    profile = state["agent_profile"]
    sq = state["sub_question"]
    # Agent 自行判断：此问题在我的领域内吗？
    if not is_relevant(sq, profile):  # LLM 快速判断
        return {"evidences": [], "skipped": True}
    # 执行研究...

优点

  • 不漏 — 全量覆盖
  • Plan 节点无需改动
  • Agent 自治性强

缺点

  • 4x LLM 调用 — 每个Agent都要判断一次
  • 大量"跳过"浪费token和延迟
  • 并行度虚高 — 实际有效的Agent少
C

方案C：关键词规则路由（简单确定性）

基于 search_queries 中的关键词（arxiv、github.com、paper、docs 等）做规则匹配，确定目标 Agent。

ROUTING_RULES = {
    "paper": ["arxiv", "paper", "论文", "doi", "conference"],
    "github": ["github", "repo", "源码", "implementation"],
    "blog": ["blog", "tutorial", "实践", "对比"],
    "docs": ["docs", "documentation", "官方", "API reference"],
}

def route_by_keywords(search_queries):
    types = set()
    for q in search_queries:
        for st, keywords in ROUTING_RULES.items():
            if any(kw in q.lower() for kw in keywords):
                types.add(st)
    return list(types) or ["blog"]  # 默认blog

优点

  • 零token开销
  • 可预测、可调试
  • Plan 节点无改动

缺点

  • 关键词覆盖不全 — 容易漏分
  • 维护成本随问题多样性增长
  • 无法处理模糊问题

推荐

方案 A — Plan 节点直接分类。
理由：Plan 节点已在调用 LLM，让 LLM 同时输出 source_type 几乎零成本。一个 sub_question 可以发多个 Agent（如"LLM Agent 架构"同时来自 paper + blog），天然支持后续 Phase 2 的交叉验证。
