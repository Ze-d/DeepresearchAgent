Human-in-the-Loop 审核节点放在工作流的哪个位置？

三种方案各有不同的控制粒度和延迟代价

A

方案A：Plan 之后审核（事前把关）

Plan 生成后暂停，人工审核研究计划和子问题分配，通过后再执行多 Agent 研究。

Plan → [HITL审核] → Research(并发) → Merge → Critique → Final

优点

  • 低成本 — 评审计划比评审全文快得多
  • 预防浪费 — 避免 Research 做错方向
  • 适合多轮迭代场景

缺点

  • 人工需要理解子问题才能有效审核
  • 审核时看不到实际结果
B

方案B：Merge 之后审核（事后把关）

多 Agent 研究完成后，所有证据汇总后暂停，人工审阅+补充，再进入 Critique 和 Final。

Plan → Research(并发) → Merge → [HITL审核] → Critique → Final

优点

  • 看到全貌再做判断 — 信息最充分
  • 可以标记不足、追加搜索
  • 可拒绝低质量 source

缺点

  • 审核材料多（sources + evidences + draft）
  • 如果审核不通过，可能浪费已完成的 Research
C

方案C：Critique 之后审核（最终把关）

完整走完 Plan→Research→Merge→Critique，最后让人审批最终报告，决定是否发布、修改或重来。

Plan → Research(并发) → Merge → Critique → [HITL审核] → Final/Re-do

优点

  • 自动化最大化 — 只在最后验证
  • Critique 提供评审参考
  • 适合对自动化质量有信心的场景

缺点

  • 如果 Critique 出问题，所有工作白费
  • 审核效率低 — 需要通读完整报告

也可以组合使用

Plan → HITL-A(可选) → Research → Merge → HITL-B(可选) → Critique → Final

支持多个审核节点，但每个节点是否启用由用户通过 API 参数控制
